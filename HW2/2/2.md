- Порушення принципу DIP (Dependency Inversion Principle). Це головна проблема. 
Клас-клієнт (MSDataBaseClient) напряму залежить від конкретної реалізації бази даних (MSDataBase). 
Вони «склеєні» намертво. За правилами SOLID, обидва класи мають залежати від абстракції (інтерфейсу), а не один від одного.
- Порушення принципу OCP (Open/Closed Principle). Код закритий для розширення. 
Якщо ми захочемо змінити Microsoft SQL на будь-яку іншу базу (наприклад, Oracle чи MySQL), нам доведеться переписувати код самого клієнта. 
Ми не можемо просто «підсунути» йому іншу базу.
- Жорстка залежність (Tight Coupling): Клієнт занадто багато знає про внутрішню логіку бази — навіть назви методів містять «MS», що робить їх не універсальними.
- Неможливість тестування 
Такий код майже неможливо протестувати ізольовано (юніт-тестами), бо ми не можемо замінити реальну базу даних на «фейкову» (mock-об'єкт) без переписування клієнта.
